## Context

当前项目是一个全新的 Python 数据生成工具项目，没有遗留代码。经实测，京东等电商平台对搜索/列表页实施了严格反爬策略（302 重定向封锁），实时抓取方案不可行。因此采用**预置静态商品库**方案：手工构建一个包含数百条参考真实市场行情商品（品牌、品名、价格）的 JSON 数据文件作为数据基座，生成器基于该库采样并在运行时补充模拟字段。

技术栈：Python 3.8+，核心依赖为标准库（random, json, uuid, datetime, pathlib, argparse）。可选依赖：pyyaml（YAML 配置增强）、kafka-python（Kafka 输出模式）。

约束：
- 数据需符合中文电商习惯（中文姓名、地址、商品名等）
- 商品基础信息（名称、品牌、价格、分类）来源于预置静态商品库
- 数据输出支持两种模式：file（JSON 文件，完全离线）和 kafka（推送到 Kafka topic）
- 订单生成器依赖商品生成器的输出
- 输出模式通过命令行参数 `--output` 判断：路径字符串为文件模式，`kafka://host:port/topic` 格式为 Kafka 模式

## Goals / Non-Goals

**Goals:**
- 构建一个仿真实的中文电商商品库 JSON 文件，涵盖 8-10 个大品类，每个品类 20-50 个商品
- 实现商品数据生成器，从商品库采样并补充模拟字段（库存、销量、描述），支持 file 和 kafka 两种输出模式
- 实现订单数据生成器，可引用已有商品数据生成完整订单，支持 file 和 kafka 两种输出模式
- 提供随机化工具模块，封装中文数据字典和随机生成逻辑
- 提供 Kafka 工具模块，封装生产者连接、批量推送、连接管理
- 提供 YAML 配置文件，统一管理生成参数和输出模式
- 商品和订单数据具备真实感和多样性

**Non-Goals:**
- 不涉及数据库写入/ORM
- 不提供 REST API 或 Web 界面
- 不生成图片文件（仅生成图片 URL 字符串）
- 不实现数据校验/清洗功能
- 不生成用户/会员体系的独立数据
- 不做网络抓取（商品库是静态预置的）

## Decisions

### 1. 模块拆分：五文件 + 一数据

**决策**：分为 `data/product_library.json`（预置商品库）、`data_utils.py`（公共工具）、`kafka_utils.py`（Kafka 推送）、`product_generator.py`（商品采样生成）、`order_generator.py`（订单生成）。

**理由**：商品库数据与生成逻辑分离，Kafka 逻辑独立封装便于复用和维护。两个生成器通过统一的 `--output` 参数即可切换 file/kafka 模式，无需分别实现。

### 2. 双输出模式设计：file 与 kafka

**决策**：通过 `--output` 参数的格式自动判定输出模式：
- 普通路径字符串（如 `output/products.json`）→ **file 模式**：输出 JSON 数组到文件（现有行为）
- `kafka://host:port/topic` URI 格式 → **kafka 模式**：逐条发送消息到指定 Kafka topic，同时可选地生成 JSON 文件作为本地备份

kafka 模式下每条数据记录序列化为 JSON 字符串作为 Kafka message value，key 使用 product_id / order_id。

**理由**：
- 单参数切换，设计简洁，用户无需记住 `--mode` 之类的额外参数
- 一次运行可以同时推 Kafka 和写本地文件（方便后期离线分析）
- 依赖 `kafka-python` 库，`kafka_utils.py` 在运行时才尝试 import，kafka 模式不可用时给出明确提示

**备选方案**：`--mode file|kafka` + `--kafka-broker` 分离参数 — 参数多但更明确，但我们的方案更简单

### 3. Kafka 推送策略

**决策**：
- 每条数据记录生成后**立即发送**（非攒批），确保生成进度可见
- 在 Kafka 模式下，支持 `--kafka-batch-size` 控制每批次发送条数（默认 1 即逐条），提升吞吐时可调大
- 使用 `kafka.KafkaProducer` 的异步发送 + `flush()` 确保全部发送完毕再退出
- 发送失败时打印错误并重试 3 次，超过则跳过该条继续
- topic 不存在时 Kafka 默认 auto-create（可配置关闭）

**理由**：数据生成工具的重点是可靠地输出数据，不可能一边生成一边断线丢失大量数据。逐条发送确保错误隔离。

**备选方案**：生成完后批量推 — 如果生成失败，Kafka 中完全无数据，不够好

### 4. 预置商品库设计

**决策**：一个 JSON 数组，每个元素包含以下预置字段：

```json
{
  "name": "iPhone 15 Pro Max 256GB 原色钛金属",
  "brand": "Apple",
  "category": "电子产品",
  "sub_category": "手机",
  "price_min": 8999.00,
  "price_max": 11999.00,
  "image_url": "https://picsum.photos/seed/iphone15pm/400/400",
  "keywords": ["5G手机", "旗舰", "大屏", "拍照"]
}
```

**理由**：
- `price_min`/`price_max` 提供价格区间，生成时随机取值，模拟不同渠道/时间点的价格波动，更加真实
- `keywords` 用于描述文案拼接和分类标记
- `image_url` 预留占位图格式，后续可替换为真实 CDN 地址
- 每个品类预置 20-50 个商品，总共 200-500 条基础数据，足以支撑万级商品采样（通过价格微调 + 同品牌不同型号变体扩展）

**备选方案**：硬编码在 Python 字典中 — 数据量大时 py 文件臃肿难维护；CSV — 不便于嵌套结构。

### 3. 商品生成策略：采样 + 变体扩展

**决策**：当请求数量超过商品库大小（如库中 300 条、请求 1000 条）时，不仅重复采样，同时通过以下方式产生变体：
- 同款不同规格（容量/颜色/尺码变体）：名称追加 "128GB版"/"256GB版" 等后缀
- 价格在 `price_min`~`price_max` 区间随机波动 ±5%
- 每个变体独立分配 product_id

**理由**：模拟真实电商中同一 SPU 下有多个 SKU 的场景，使生成数据更自然。

### 4. 商品分类体系

**决策**：预置 8-10 个一级分类，每个分类下 3-5 个细分子类：

| 大类 | 子类示例 | 商品库条数 |
|------|---------|-----------|
| 电子产品 | 手机、笔记本电脑、平板、耳机、智能手表 | 40-50 |
| 服装 | 男装、女装、童装、内衣、运动服饰 | 40-50 |
| 食品饮料 | 休闲零食、饮料冲调、生鲜水果、粮油调味 | 30-40 |
| 家居生活 | 家纺、家具、厨具、收纳用品、灯具 | 30-40 |
| 美妆个护 | 护肤、彩妆、洗护、口腔护理、男士护理 | 30-40 |
| 母婴 | 奶粉辅食、纸尿裤、童车童床、孕产用品 | 20-30 |
| 运动户外 | 运动鞋、健身器材、户外装备、骑行 | 20-30 |
| 图书文娱 | 小说、教辅、少儿图书、电子书、文具 | 20-30 |

注：以上为初始规划，实际商品库按此结构填充。价格区间参考京东/天猫同品类实际价格带。

### 5. 订单-商品关联方式：引用商品 ID

**决策**：订单生成器读取 `products.json`，通过商品 ID 引用关联商品，订单中存储商品快照（含下单时的价格和数量）。

**理由**：模拟真实电商场景——订单创建时记录当时的商品价格快照，而非实时查询。每个订单包含 1-5 个商品明细行。

### 6. JSON 输出格式

**决策**：输出为 JSON 数组，每个元素为一条记录，顶层无额外包装。商品和订单分别输出到独立文件。

**理由**：JSON 是最通用的数据交换格式。

### 7. 配置方式：YAML + 命令行参数（JSON 兜底）

**决策**：优先读取 `config.yaml`，如果 pyyaml 未安装则回退到 JSON 配置文件。命令行参数可覆盖配置文件中的值。

**理由**：YAML 可读性好，但避免强制外部依赖。

## Risks / Trade-offs

- **[商品库维护成本]** 预置商品库需人工构建初始数据（200-500 条）→ 一次性投入，建好后长期复用；按真实市场行情填写，质量高于随机生成
- **[数据真实性上限]** 预置库的商品名仍然是人造的，不如真实京东数据→ 但通过参考京东真实商品名和价格区间填写，实际仿真度足够测试用途
- **[数据量性能]** 生成十万级数据时可能较慢 → 10 万级数据在 Python 中可接受（预估 <30 秒）；如需更大量可后期引入批处理写入
- **[随机种子一致性]** 每次运行产出的数据不同 → 提供 `--seed` 参数支持固定随机种子以复现结果
- **[变体自然度]** 自动生成的规格变体可能不如手工撰写的自然→ 为每个商品库条目预置一组可选规格后缀（如 "128GB/256GB/512GB"），而非随机拼接字符串
- **[Kafka 依赖]** Kafka 模式需要额外的 `kafka-python` 依赖和可用的 Kafka Broker → `kafka-python` 作为可选依赖，在 `kafka_utils.py` 中延迟导入；无安装时 Kafka 模式会给出明确提示
- **[Kafka 连接失败]** Broker 不可达时所有数据丢失 → 同时写本地 JSON 文件作为备份；连接失败时立即报错退出，不静默丢弃
